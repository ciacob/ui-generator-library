# ui-generator-library
Library that uses AS3 reflection to build a basic UI made out of (Flex) components
that bind back to reflected class' respective public members. A number of supported
custom annotations make the resulting UI more flexible.

Note that the compiler must be forced to retain those custom annotations. This is
achievable by adding the compiler argument:

-keep-as3-metadata+=Annotation1,Annotation2,AnnotationN

In Flash Builder, this will be added under 
"Properties"->"Flex Compiler"->"Additional compiler arguments".


	/**
	 * OVERVIEW
	 * Inspects a given object (the `originator`) and generates UI controls which it 
	 * then adds as children to a given container (the `container`).
	 * 
	 * User interactions on the generated UI are automatically observed and 
	 * translated into value changes of the originator`s class members. Programmatic
	 * changes in the originator are also observed, and cause the generated controls
	 * to update their values. Editable controls will ignore these updates 
	 * while they are focused, and only accept user input.
	 * 
	 * SUPPORTED SCENARIOS
	 * The class will succeed in generating UI if:
	 * - The `originator` is a custom type, and it presents at least one getter/setter
	 *   pair (aka readwrite "accessor") of one of the supported types. List of choices
	 * 	 also require a third element, a getter (readonly accessor) named [SAME_NAME]Src.
	 * 
	 * SUPPORTED TYPES
	 * Supported types are:
	 * - Boolean
	 * 	 Generated as a checkbox.
	 * 
	 * - String
	 * 	 Generated as an input text box.
	 * 
	 * - Number, Integer or Unsigned Integer
	 * 	 Generated as either a  numeric stepper or a slider, based on additional metadata.
	 * 	 Decimals will be accepted or not based on accessor type and additional metadata.
	 * 
	 * - Object
	 * 	 Generated as a single selection list; a readonly accessor named [SAME_NAME]Src
	 *   is required to provide the options for the list. 
	 * 
	 * - Array
	 * 	 Generated as a multiple selection list; a readonly accessor named [SAME_NAME]Src
	 *   is required to provide the options for the list. 
	 */


=====
NOTE:

- se pare ca variabilele publice (public var gigi : String = "Gigi";) nu sunt recunoscute si nu sunt generate. Doar perechile "set/get";

- pentru orice pereche de setter/getter (numite in continuare "accessori") de tipul Number, int sau uint, se pot seta doua metadata, "Minimum" si "Maximum" (Ex: [Minimum(value="3")] public function set myNumber...) care definesc limitele intre care poate fi setata valoarea numerica. Daca una din limite lipseste, se pune automat "int.MIN_VALUE" -- adica `-2147483647` -- sau "int.MAX_VALUE" -- adica `2147483647`; Daca minima si maxima au fost setate la `0` si `1`, respectiv, se randeaza un Slider; altfel, se randeaza un Numeric Stepper.

- biblioteca de cod nu se ocupa de layout-ul elementelor generate; el poate fi setat direct folosind elementul "parinte" (numit si "target" sau "container" in API-ul bibliotecii); daca containerul nu ofera aceasta facilitatea, se poate crea un "builder" care seteaza coordonatele fiecarui element generat, ca alternativa. La fel stau lucrurile si cu fonturile folosite, stilul label-urilor, etc.

- in mod normal, accesorii vor avea nume scrise in "camelCase", care va fi rezolvat la "Title Case" (doar prima litera se capitalizeaza, celelalte se pastreaza asa cum sunt). Pentru a exprima corect o propozitie se poate face uz de "underscores" pentru a separa cuvintele: "tick_the_checkbox_if_you_are_John" devine "Tick the checkbox if you are John". Daca se utilizeaza cifre in numele de accesor, ele trebuie separate prin "underscores", altfel se vor uni cu cuvantul anterior: "myObject2" devine "My Object2", dar "myObject_2" devine "My Object 2".

- accesorii sunt raportati in ordine alfabetica (dupa numele lor in varianta "decamelized"); daca nu se doreste asta, se poate adauga metadata-ul "Index" ([Index(value="0")] public function set myNumber ...).

- metadata-ul "Description" se poate folosi pentru a adauga o descriere elementului respectiv: [Description(value="Sit adipiscit elit")]. Cu builder-ul Flex implicit, descrierea este setata ca "tooltip" pe label-ul elementului generat.

- pentru a crea un ComboBox e nevoie de trei elemente adaugate in clasa sursa: o pereche get/set de tip Object cu numele dorit, Ex.: "MyChoice", si un getter de tip Array cu numele terminat in "Src", Ex.: "MyChoiceSrc". Fara getter, ComboBox-ul o sa apara gol. Array-ul de sursa poate sa contina orice tip primitiv de date sau Obiecte. Daca contine Obiecte, fiecare item (Obiect) trebuie sa contina un camp "label" cu o valoare de tip String, care va fi afisata in lista ComboBox-ului.

- NOTA: daca "originator"-ul este de tip Obiect simplu, Array-ul de sursa trebuie sa contina obligatoru Obiecte DACA SE DORESTE POPULAREA LISTEI CU O VALOARE IMPLICITA. Daca contine alt tip de valori, ex., string-uri, setarea proprietatii (de tip Object) care reprezinta lista ii va schimba tipul de la Object la String, iar campul va fi randat ca un camp de text.

- [DOAR DACA SE FOLOSESTE UiFlexBuilder]: daca o terta-parte schimba valoarea unui membru public pentru care a fost generat un component vizual (UI component), valoarea se schimba si in componentul vizual corespunzator. Pentru ca acest comportament sa functioneze, membrul public trebuie sa aiba metadata-ul "Bindable": (Ex: [Bindable] public function set myNumber...). De asemenea, pentru componentele vizuale care pot folosi Obiecte ca optiuni disponibile (ComboBox si List), binding-ul va functiona doar daca terta-parte seteaza valoarea membrului public LA INSTANTE DE OBIECTE din sursa originala. Utilizarea de Obiecte identice ca continut DAR DIFERITE CA INSTANTA nu va functiona (Lista sau ComboBox-ul va sterge selectia, ca si cum membrul public ar fi fost setat la `null`).

- NOTA: functionalitatea descrisa mai sus NU ESTE DIPONIBILA cand originator-ul este un Obiect simplu (ex.: var myObject = {myValue : 7}).

- accesorii publici din clasa de origine al caror nume incepe cu "$" NU vor fi randati.

- generarea va avea loc si fara metadata aditionale, dar cu anumite limitari:
  -- daca lipseste "Index", ordinea in care sunt generate componentele vizuale este arbitrara;
  -- daca lipseste "Description", elementul vizual generat nu va avea documentatie/sugestii de utilizare/tooltip;
  -- daca lipsesc "Minimum" si/sau "Maximum", elementul vizual generat va fi un camp "NumeriStepper" cu o plaja foarte larga de operare (intre -2147483647 si 2147483647). In plus, nu se pot obtine elemente de tip "Slider", intrucat un "Slider" este generat doar pentru accesori de tip Number cu "Minimum" egal cu `0` si "Maximum" egal cu `1`.

- If a description is given for an accessor, it can use the special syntax [accessorName] to make a reference to another (related) accessor. We expand these references to the accessor's "friendly" name.

-- metadata-urile pot fi folosite doar pentru cazul cand "originator"-ul este instanta unei clase; cand este un Obiect, rolul metadata-urilor este substituit de o proprietate numita "uiGeneratorConfig", de tip Obiect, care poate fi adaugata in originator. Ea va accepta ca chei numele proprietatilor care se doresc configurate, iar ca valori, Obiecte care vor contine aceleasi chei ca si metadata-urile aditionale (ex.: "Index", "Description", "Minimum", "Maximum"), cu aceleasi valori si aceleasi functii ca si metadata-urile aditionale. Proprietatea "uiGeneratorConfig" nu poate fi folosita in cazul instantelor de clase.

Exemplu:
			private const _sourceObject : Object = {
				myNumber : 45.2,
				myOtherNumber : 0.00001,
				myInteger: 3,
				myBoolean : true,
				myObject : {},
				myObjectSrc: ['red', 'green', 'blue'],
				myObject2 : {},
				myObject2Src: [{label: 'small', value: 1}, {label: 'medium', value: 3}, {label: 'large', value: 5}],
				myArray : [],
				myArraySrc : [{label: 'Bach', value: 3}, {label: 'Haydn', value: 2}, {label: 'Mozart', value: 1}],

				// Provide more configuration fo the UI components to be generated
				uiGeneratorConfig : {
					myNumber: { Index: 1 },
					myOtherNumber : { Index: 1, Minimum: 0, Maximum: 1},
					myInteger: { Index: 3 },
					myBoolean : { Index: 4 },
					myObject : { Index: 5 },
					myObject2 : { Index : 6 },
					myArray: { Index : 7 }
				}
			};

- elementele generate se pot accesa folosind "<<container>>.getchildByName(<<accessor_name>>)" sau "<<ui_generator_instance>>.getComponentByName(<<accessor_name>>)". a Doua modalitate este mai rapida decat prima.

- se pot folosi si componente custom pentru randarea accesorilor dintr-o clasa sau a proprietatilor dintr-un Obiect. In cazul claselor, se foloseste metadata-ul aditional "CustomComponent". Evident, si "builderul" utilizat trebuie sa ofere functionalitatea de creare a unui component custom. Accepta un numar variabil de chei, dintra care una singura este necesara, anume "classFqn". Cheile sunt descrise in tabelul de mai jos:

-- classFqn: trebuie sa contina numele complet calificat al clasei componentului respectiv, ex: "com.example.components.MyComponent"; clasa respectiva trebuie sa fi fost in prealabil inclusa si compilata in clasa care contine containerul in care se genereaza; aceasta se poate face prin declararea unei variabile de tipul respectiv, ex.: "private var _compiler_helper : MyComponent;" in cazul in care clasa componentului nu este disponibila, se trece tacit la urmatorul accesor din lista (eroarea apare doar in modul Debug, in consola);

-- hideLabel: optional, presupus "false"; se va seta pe "true" daca este nevoie sa fie ascuns "Label"-ul implicit care se adauga inainte de orice component generat;

-- dataproviderName: optional, presupus "dataProvider"; se va seta la numele proprietatii care alimenteaza componentul custom cu date; de exemplu, intr-un component de tipul "List" standard, aceasta proprietate ar fi "dataProvider"; intr-un component "NumericStepper" standard, aceasta proprietate ar fi "value";

-- bindingSource: optional, defaults to the accessor name; alegerea unei proprietati alternative ca sursa de binding poate fi uneori necesara, atunci cand componentul este de tip lista si/sau implica functionalitatea de "selectie", iar tipul necesar pentru popularea selectiei este diferit de tipul de date al accesorului. In acest caz, in clasa "originator" se poate implementa un setter (marcat ca "[Bindable]") care sa primeasca datele convertite in formatul asteptat de "selectie". Binding-ul este o functionalitate disponibila doar cand se foloseste Flex;

-- bindingTarget: optional, defaults to the components's dataprovider property, either default or custom defined; alegerea unei proprietati alternative ca target de binding este necesara in cazul componentelor de tip lista (sau) care implementeaza conceptul de selectie. In acest caz, doua seturi sunt mentinute, setul cu optiunile disponibile si setul cu cele "selectate". In acest caz, oricare din cele doua seturi reprezinta un target plauzibil pentru binding; prin utilizarea proprietatii `bindingTarget` se poate stabili care dintre cele doua seturi sa fie modificate cand un Event de binding este difuzat. Componentele standard de tip lista (ComboBox si List) fac binding pe selectie, nu pe optiunile disponibile. Acestea pot fi schimbate manual daca e nevoie.  Binding-ul este o functionalitate disponibila doar cand se foloseste Flex; daca componentul nu are o sursa de tip Array (numita <<accessor_name>>Src), inseamna ca nu implementeaza selectie de elemente; in aceasta situatie, "bindingTarget" va fi ignorat chiar daca e setat, si proprietatea dataprovider va fi utilizata, fie cea default sau customizata.

-- changeEventName: optional, presupus "change" (sau Event.EventChange); daca este setat, trebuie sa contina numele Event-ului pe care componentul il difuzeaza cand starea lui interna se schimba in urma actiunii utilizatorului. Daca componentul nu implementeaza selectie, atunci interactiunea rezulta in modificarea proprietatii accesibile prin "dataproviderName". Daca implementeaza selectie, atunci interactiunea rezulta in modificarea proprietatii accesibile prin "bindingTargetName". Oricare ar fi situatia, un Event cu numele definit de "changeEventName" trebuie sa fie difuzat de component in una din aceste situatii pentru ca actiunea utilizatorului asupra componentului generat sa se reflecte asupra accesorului corespondent din clasa "originator".

Orice alta cheie, daca e prezenta, va fi setata ca atare, cu valoarea aferenta, pe componentul nou-creat.
